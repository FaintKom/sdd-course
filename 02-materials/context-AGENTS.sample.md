> Задаёт агенту архитектуру, конвенции и границы, чтобы он не трогал лишнее и не додумывал контекст.

## Проект
<2–3 предложения: что это за продукт, на каком стеке, какая среда выполнения и версии.>

## Архитектура (карта)
- **Точки входа:** `<src/main..., cmd/...>`
- **Ключевые модули:** `<модуль → назначение>`
- **Слои / границы:** `<api / domain / infra — кто от кого зависит>`
- **Данные:** `<БД, миграции, где схема>`

## Конвенции (как принято в этом репо)
- Стиль/линтер: `<ESLint / Ruff / gofmt ...>` — следовать существующему
- Именование: `<правила>`
- Тесты: `<раннер, где лежат, как запускать>`
- Коммиты/ветки/PR: `<правила команды>`

## Границы для агента (КРИТИЧНО)
- **НЕ менять:** `<пути / файлы / модули>` (генерённые, легаси-ядро, конфиги прод)
- **НЕ трогать без явного запроса:** публичные API, схему БД, CI-конфиги
- **НЕ добавлять зависимости** без разрешения в спеке
- **Менять только в рамках текущей задачи** (scope из спеки)

## Как агент должен работать
- Сначала прочитать спеку задачи, потом действовать в её рамках
- Перед крупными изменениями — показать план (шаги + затронутые файлы)
- Минимальный diff: не рефакторить попутно то, о чём не просили
- Если контекста не хватает — спросить, а не додумывать

## Формат результата
- diff/PR по одной задаче, описание + ссылка на спеку
- Тесты на новое поведение
- Список изменённых файлов с обоснованием каждого
