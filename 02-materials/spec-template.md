# Шаблон спецификации задачи (Spec)

> Артефакт недели **W2 «От задачи к спецификации»**. Студент заполняет на свою задачу
> сквозного проекта. Спека — это ТЗ для AI-агента: она удерживает его в рамках.
> Критерий готовности спеки: заполнены все 5 секций; по ней можно проверить результат.

---

## 0. Метаданные
- **Задача / тикет:** <ссылка или ID>
- **Тип:** фича / багфикс / рефакторинг
- **Автор спеки:** <имя>
- **Целевой репозиторий + ветка:** <repo @ branch>

## 1. Цель и контекст (зачем)
Одно-два предложения: какую пользователю/бизнесу проблему закрываем. Не «как», а «зачем».
> Пример: «Пользователь не может отменить заказ после оплаты — растёт нагрузка на поддержку.
> Нужна самостоятельная отмена в окне 30 минут.»

## 2. Требования (что должно быть сделано)
Список наблюдаемого поведения. Каждый пункт проверяем.
- [ ] <требование 1>
- [ ] <требование 2>

**User story (опц., для middle+):**
`Как <роль>, я хочу <действие>, чтобы <ценность>.`

**Acceptance criteria в нотации EARS** (стандарт индустрии, тестируемо — см. бенчмарк):
- `WHEN <событие> THE SYSTEM SHALL <поведение>`
- `IF <условие>, THEN THE SYSTEM SHALL <поведение>`
- `WHILE <состояние> THE SYSTEM SHALL <поведение>`
> Пример: `WHEN пользователь жмёт «отменить» в окне 30 мин THE SYSTEM SHALL отменить заказ и вернуть оплату.`

### Clarification gate (до передачи агенту)
Помечай всё неясное `[NEEDS CLARIFICATION: вопрос]` и разрешай ДО планирования/реализации.
Спека с незакрытыми `NEEDS CLARIFICATION` в работу не идёт.

## 3. Ограничения и границы (чего агент НЕ делает)
Самая важная секция против «агент сносит рабочее решение». Явно:
- **Не менять файлы:** `<пути / модули>`
- **Не трогать:** <публичные API, схему БД, конфиги прод>
- **Технические ограничения:** <язык, версии, паттерны, без новых зависимостей и т.п.>
- **Scope:** только <X>. Всё за рамками — отдельной задачей.

## 4. Критерии готовности (Definition of Done)
Бинарно проверяемо. «Принято», когда выполнены ВСЕ:
- [ ] Поведение из секции 2 реализовано
- [ ] Тесты из секции 5 проходят (CI зелёный)
- [ ] Ограничения секции 3 соблюдены (diff не выходит за scope)
- [ ] Ревью по чек-листу пройдено (см. `review-checklist-ai-code.md`)
- [ ] PR оформлен: описание + ссылка на эту спеку

## 5. Тесты / приёмочные сценарии
Как проверим. Прямой happy-path + негативы + edge.
```
Сценарий 1 (happy): дано <...> когда <...> тогда <...>
Сценарий 2 (негатив): дано <...> когда <...> тогда <ошибка/отказ>
Сценарий 3 (edge): <...>
```
Для middle+/T3 — формат Given-When-Then (Gherkin).

---

## Чек-лист самопроверки спеки (перед передачей агенту)
- [ ] Цель отвечает на «зачем», не на «как»
- [ ] Каждое требование наблюдаемо и проверяемо
- [ ] Есть явный список запретных файлов/зон
- [ ] DoD бинарный (можно сказать да/нет по каждому пункту)
- [ ] Есть хотя бы 1 негативный и 1 edge-сценарий
- [ ] По спеке третий человек поймёт, что и в каких рамках делать

---

## Тиры (W2)
- **T1 (junior+):** заполнить по worked example, happy-path фича, секции 1–4.
- **T2 (middle):** своя задача, все 5 секций, прогнать агента с/без спеки и зафиксировать разницу.
- **T3 (senior/lead):** Gherkin + ADR-ссылка, спека на легаси-кейс, годная как командный стандарт.

## Учебное упражнение «с/без спеки»
1. Прогнать агента на задаче БЕЗ спеки → сохранить diff_0.
2. Прогнать ту же задачу СО спекой → diff_1.
3. Зафиксировать разницу: лишние файлы, выход за scope, недостающее поведение.
   Это germane-нагрузка: строит схему «спека управляет агентом».
