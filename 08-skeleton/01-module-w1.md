## Урок 1.1 — Почему промпты ломаются: 4 причины сбоя агента
**Тип:** асинхрон, лонгрид 12–15 мин (текст + скрины кейсов) + квиз 5 мин. **4C/ID: SI** — ментальная модель сбоев, на ней
держится весь курс (каждая следующая неделя = контрмера одному типу сбоя).

**Микро-ОР:** студент называет 4 типа сбоя агента и отличает их друг от друга на чужом примере.

**Тезисный контент (порядок подачи):**
1. Хук: реальный диалог «сделай мне фичу» → агент уверенно выдал не то. Не анти-AI: модель сильная, сбой — в управлении. *[SME-slot: свой провальный диалог, скрин]*
2. Рамка: «вайбкодинг vs управляемая разработка» — разница не в инструменте, а в наличии процесса.
3. Тип 1 — **контекст**: агент не знает проекта (архитектуру, конвенции, границы) → лезет не туда. Симптомы: трогает чужие файлы, дублирует существующее.
4. Тип 2 — **доверие**: агент додумывает недосказанное (галлюцинации API, выдуманные требования). Симптомы: «откуда он это взял?»
5. Тип 3 — **хаос**: нет плана и контрольных точек → scope creep, переписал полпроекта. Симптомы: diff на 40 файлов вместо 3.
6. Тип 4 — **качество**: код «работает», но без тестов, с тех-долгом, небезопасный. Симптомы: ревью пропущено, падает на проде.
7. Карта курса: W2–W3 лечат контекст и доверие, W4 — хаос, W5–W6 — качество, W7–W8 — всё вместе. (Прозрачность = мотивационная поддержка.)
8. Тезис-якорь курса: **непредсказуемость ИИ = нечёткая постановка задачи**.

**Мини-практика (квиз, автопроверка):** 5 мини-кейсов (скрин диалога/диффа + 3–6 строк описания) →
выбрать тип сбоя. Кейс 5 — смешанный (контекст+хаос), допускает 2 ответа: учит, что типы сочетаются.
Критерий: ≥4/5. При ошибке — пояснение к каждому варианту (формирующая обратная связь, не «неверно»).
*[SME-slot: кейсы из реальной практики; синтетические допустимы, но скрины должны выглядеть живыми]*

**Материалы к производству:** лонгрид (тезисы выше → текст), 5 кейсов-скринов, квиз в LMS.

**Связи:** ← pre-work (окружение, кандидат темы). → 1.2 (что с этим делать), 1.3 (кейсы воркшопа —
та же рамка, сложнее), W2+ (каждая неделя ссылается на свой тип сбоя).

**Тиры:** контент один для всех; T2/T3 — доп. вопрос «вспомни свой случай каждого типа» (вход в 1.2).

## Урок 1.2 — Ментальная модель «менеджер отделов»: доверить vs проверять
**Тип:** асинхрон, лонгрид 8–10 мин + мини-практика 15 мин. **4C/ID: SI** — вторая опорная модель курса
(«как думает эксперт», когнитивная стратегия распределения работы).

**Микро-ОР:** студент распределяет типовые задачи разработки по зонам «доверить / проверить выборочно / не отдавать», аргументируя ценой ошибки.

**Тезисный контент:**
1. Метафора: ИИ-агент = сильный новый сотрудник без контекста компании. Менеджер не пишет код за отдел — он ставит ТЗ, задаёт границы, принимает работу.
2. Критерий распределения — **цена ошибки × проверяемость**: дёшево проверить и дёшево ошибиться → доверяй; дорого ошибиться (security, деньги, данные, миграции) → контролируй или не отдавай.
3. Зона «доверить»: boilerplate, тесты по образцу, рефактор по правилам, документация, разовые скрипты. *[SME-slot: свой список + где граница сдвинулась за последний год]*
4. Зона «проверить»: бизнес-логика, интеграции, всё на границах систем.
5. Зона «не отдавать» (пока): архитектурные решения, security-критичное, работа с прод-данными.
6. Границы не догма: смещаются с ростом твоего навыка контроля (а не «смелости»). К W8 студент пересмотрит карту — это заложено в рефлексию курса (8.5).
7. Анти-паттерны: «доверяю всё» (= тип сбоя «доверие»), «проверяю каждую строку» (= ИИ бесполезен).

**Мини-практика (сортировка, автопроверка + эталон):** 8 задач (фикс опечатки в README; миграция
БД; юнит-тест по образцу; эндпоинт с оплатой; рефактор переименования; парсер CSV; правка CORS;
выбор библиотеки авторизации) → разложить по 3 зонам. Эталон с пояснением **почему** (цена ошибки ×
проверяемость); 2 задачи намеренно спорные, эталон показывает «зависит от…».
Критерий: совпадение ≥6/8 ИЛИ осмысленный аргумент в свободном поле (ручная выборочная проверка).

**Материалы к производству:** лонгрид, интерактив-сортировка (или форма), эталон с пояснениями.

**Связи:** ← 1.1 (типы сбоя = что бывает, если зону перепутал). → 1.3 практика «карта
доверия/контроля» (там — та же сортировка, но для **своей** будущей задачи), 1.4 ДЗ.

**Тиры:** T1 — список задач дан; T2 — добавь 3 задачи со своей работы; T3 — набросай эвристику для
своей команды (1 абзац, вход в T3-ветку воркшопа).

## Урок 1.4 — ДЗ: карта доверия/контроля + тема проекта
**Тип:** практика проекта, 1–2 ч. **4C/ID: LT** — первый шаг сквозного проекта.

**Формулировка для студента:**
1. Дозаполни карту доверия/контроля (`../03-lesson-01/trust-control-map-template.md`) **для своей
   будущей задачи** — не абстрактно.
2. Зафиксируй тему сквозного проекта: фича / багфикс / рефакторинг. Опиши 3–5 предложениями: что,
   где (репо/стек), как поймёшь, что готово.

**Критерий «принято» (дуальная оценка):** тема валидна (изменение в коде, а не «приложение с нуля»;
реалистична на 8 недель по 2–4 ч); карта заполнена связно (зоны аргументированы ценой ошибки).
Возврат на доработку — с указанием, какой пункт.

**Тиры:** T1 — тема из готового списка (учебный репо-шаблон, 6–8 тем). T2 — своя рабочая задача.
T3 — задача в легаси / командный контекст.

## Урок 1.5 — Рефлексия + мост к W2
**Тип:** асинхрон, 15 мин. **4C/ID:** — (метакогнитивный слой; закрывает оценку soft skills).

**Форма рефлексии (Гиббс, 5 вопросов в LMS):**
1. Что было на неделе? (описание, 1–2 предложения)
2. Какой момент зацепил/смутил? (чувство)
3. Мой главный тип сбоя в прошлом опыте с ИИ — какой и почему? (анализ — связывает 1.1 с личным опытом)
4. Что меняю в подходе уже сейчас? (вывод)
5. Чего жду/боюсь в работе со спекой? (план → вход в W2, и сигнал фасилитатору перед ямой #1)

Ответы видит фасилитатор (выборочно комментирует 🟡-сигналы). Не оценивается — норма blameless.

**Pre-work W2 (в этом же уроке):** прочитать пары хорошая/плохая спека
(`../02-materials/spec-template.md`); проверить окружение по чек-листу, если не успел.

**Материалы:** форма рефлексии (шаблон один на все недели — меняются вопросы 3 и 5), тексты pre-work (есть).
