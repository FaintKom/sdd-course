## Урок 3.1 — Как агент видит проект: что он знает и чего не знает
**Тип:** асинхрон, лонгрид 10–12 мин (текст + скрины diff) + квиз 5 мин. **4C/ID: SI** — ментальная модель «глаза агента»;
без неё контекст-файл пишется как ритуал, а не как инженерное решение.

**Микро-ОР:** студент объясняет, какую информацию агент имеет в момент задачи, и решает, что
класть в спеку (per-task), а что в контекст-файл (per-repo).

**Тезисный контент (порядок подачи):**
1. Хук: «агент полез в схему БД» — реальный кейс: задача была про UI, агент «улучшил» миграцию. Почему? Ему никто не сказал, что нельзя. *[SME-slot: свой кейс протечки границ, скрин diff]*
2. Контекстное окно: агент видит только то, что ему показали — открытые файлы, результаты поиска, контекст-файл. Он НЕ знает: историю решений, негласные конвенции, что «трогать нельзя», что уже пробовали и откатили.
3. Как инструменты собирают контекст: AGENTS.md/CLAUDE.md подхватываются автоматически в начале сессии — это «вводный инструктаж нового сотрудника» (мостик к модели «менеджер отделов» из 1.2).
4. Разделение ответственности: **спека = задача (меняется каждый раз), контекст-файл = репозиторий (постоянный)**. Правило сортировки: информация переживёт эту задачу? → контекст-файл.
5. Связь с типами сбоя (1.1): контекст-файл — прямая контрмера сбою «контекст». Возврат к слайду.
6. Экономика: пишется один раз, удешевляет каждую следующую задачу — в W4–W8 студент будет переиспользовать этот файл каждую неделю.
7. Анти-тезис: контекст-файл не магия — границы «на словах» протекают; проверка только боем (прогоном). Это тема воркшопа.

**Мини-практика (квиз, автопроверка):** 5 ситуаций → «куда положить: спека / контекст-файл / никуда
(агент сам увидит)»: запрет менять схему БД (контекст); AC текущей фичи (спека); стиль именования
в репо (контекст); список изменяемых файлов задачи (спека); содержимое package.json (никуда — сам
прочитает). Критерий ≥4/5, пояснения через правило «переживёт ли задачу».

**Материалы к производству:** лонгрид, квиз.

**Связи:** ← 2.5 pre-work (карта репо с запретными зонами — сырьё), 1.2 («менеджер» → инструктаж).
→ 3.2 (структура файла), 3.3 (пишем вживую), W4 (связка спека+контекст идёт в работу агента).

**Тиры:** общий; T3 — вопрос «что из этого масштабируется на монорепо/несколько команд».

## Урок 3.2 — Анатомия AGENTS.md/CLAUDE.md: архитектура, конвенции, запретные зоны
**Тип:** асинхрон, лонгрид 10 мин + мини-практика 15 мин. **4C/ID: PI** — структура файла по
секциям, типовые дыры, инструментная специфика.

**Микро-ОР:** студент критикует готовый контекст-файл — находит дыры в границах до того, как их
найдёт агент.

**Тезисный контент (по секциям образца `context-AGENTS.sample.md`):**
1. Шесть секций образца: Проект → Архитектура → Конвенции → **Границы (КРИТИЧНО)** → Как работать → Формат результата. Зачем каждая, по 1–2 предложения.
2. Секция «Границы» — ядро: запреты с **конкретными путями** (`src/legacy/`, `migrations/`), не «аккуратнее с базой». Агент исполняет пути, не намёки.
3. Типовые дыры границ (из практики, разбираются на воркшопе): забыли тесты и CI-конфиги; забыли «не добавлять зависимости»; нет «минимальный diff, не рефакторить попутно».
4. Антипаттерн «простыня»: 300 строк контекста — агент теряет важное в шуме. Краткость = управляемость; не дублировать то, что агент сам прочитает (package.json, структуру папок).
5. Антипаттерн «мёртвый файл»: контекст устарел → агент уверенно делает по старой архитектуре. Контекст-файл живёт в PR вместе с кодом.
6. Инструментная специфика: AGENTS.md (стандарт), CLAUDE.md, `.github/copilot-instructions.md` — одна суть, разные имена. **Один формат на поток** (правило курса), остальное — справкой.
7. Принцип проверки: каждая граница тестируется прогоном («проверка боем» из чек-листа образца).

**Мини-практика «критика файла» (15 мин, разметка + эталон):** дан AGENTS.md учебного репо
с **5 подсаженными дырами**: границы без конкретных путей («не трогать важное»); нет секции формата
результата; конвенция противоречит реальному коду репо; устаревшая архитектура (модуль переименован);
180 строк пересказа README (шум). Найти ≥3. Эталон объясняет каждую дыру через «что агент сделает
не так». *[SME-slot: дырявый AGENTS.md должен соответствовать реальному учебному репо-шаблону потока —
противоречия проверяемы по коду]*

**Материалы к производству:** лонгрид, дырявый AGENTS.md + эталон дыр (под учебный репо-шаблон).

**Связи:** ← 3.1 (зачем секции). → 3.3 (worked example «до/после»), 3.4 (свой файл по образцу).

**Тиры:** T1 — практика обязательна; T2 — найди те же 5 типов дыр в контекст-файле своего рабочего
репо (если есть) или составь список кандидатов в границы; T3 — какие правила вынес бы на уровень
организации (общий шаблон команд).

## Урок 3.4 — ДЗ: контекст-файл проекта + тест границ боем
**Тип:** практика проекта, 2 ч. **4C/ID: PI→LT** — шаг 3 сквозного проекта.

**Формулировка для студента:**
1. Доведи контекст-файл своего репо по образцу (все 6 секций; границы — конкретными путями).
2. **Проверка боем:** прогони агента на мелкой задаче (не из проекта — любая безопасная: поправить
   текст, добавить мелкий метод). Убедись: запретные файлы не тронуты, формат результата соблюдён.
   Если протекло — почини границу, прогони снова. Приложи короткий лог: «граница X — держит/протекла→исправил».
3. Подготовь связку **спека (W2) + контекст (W3)** на следующий шаг проекта — это вход в W4
   (агент пойдёт работать по ним).

**Критерий «принято»:** контекст-файл в репо, чек-лист образца пройден; в последнем прогоне агент
не правил запрещённое; связка спека+контекст готова к W4. Возврат — с указанием секции/границы.

**Тиры:** T1 — заполняет образец под учебный репо; T2 — под свой рабочий репо, проверка боем
обязательна; T3 — правила уровня монорепо/команды, переносимые (+ короткое обоснование решений).

**Материалы:** образец (есть), чек-лист в образце (есть), форма сдачи: файл + лог проверки боем.

## Урок 3.6* — Опциональный трек (frontend/fullstack): дизайн-система как спека для UI
**Тип:** асинхрон, лонгрид 15–20 мин, **вне обязательной нагрузки**. **4C/ID: SI** (расширение
модели «контекст» на UI-домен). Канон: `program-and-outcomes.md` §7 — не отдельная неделя, врезка W3–W4.

**Микро-ОР:** студент применяет принцип «контекст управляет агентом» к UI: токены, компоненты,
правила = границы для UI-генерации.

**Тезисный контент:**
1. Проблема: UI от агента «каждый раз новый» — цвета, отступы, компоненты не из системы.
2. Дизайн-система = контекст-файл для UI: токены (цвета/типографика/сетка), список готовых
   компонентов («используй `Button` из `ui/`, не пиши свой»), правила композиции.
3. Куда класть: секция «Конвенции UI» в том же AGENTS.md + ссылка на файл токенов.
4. Мини-демо «до/после»: генерация формы без правил и с правилами. *[SME-slot: пример на стеке
   потока; если SME не фронтенд — взять гостевой фрагмент]*
5. Перенос: тот же принцип для любого домена (API-гайдлайны, схема БД) — границы предметные, механизм один.

**Практика:** не обязательна; для желающих — добавить UI-секцию в свой контекст-файл (входит в их 3.4).

**Материалы к производству:** лонгрид + демо-скрины «до/после».
