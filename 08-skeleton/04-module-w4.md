## Урок 4.1 — Как агент планирует: декомпозиция, scope, контрольные точки
**Тип:** асинхрон, лонгрид 12–15 мин (текст + скрины двух прогонов) + квиз 5 мин. **4C/ID: SI** — ментальная модель ведения агента
(«вести, не запускать»); процедурный дрилл — в 4.2.

**Микро-ОР:** студент объясняет, из чего состоит agent plan, и расставляет контрольные точки
по цене ошибки шага.

**Тезисный контент (порядок подачи):**
1. Хук: два режима работы — «нажал и жду» vs «веду по плану». Один и тот же агент, одна спека; без ведения — diff на 14 файлов, с ведением — 3 файла по scope. *[SME-slot: скринкаст/скрины обоих прогонов]*
2. Связь с типом сбоя «хаос» (1.1): план + контрольные точки = контрмера. Возврат к слайду «4 типа сбоя».
3. Agent plan — три элемента: **шаги** (декомпозиция задачи), **scope** (какие файлы/модули шаг трогает), **контрольные точки** (где останавливаюсь и сверяю diff со спекой ДО продолжения).
4. Атомарность: одна задача = один PR. Раздутый scope — главная причина потери контроля; правило «если план длиннее 5–7 шагов — режь задачу».
5. Где ставить чекпоинты — по цене ошибки шага (мостик к 1.2 «цена ошибки × проверяемость»): после плана (до кода!), после шагов с риском (миграции, публичные API), перед финальным коммитом. Не «после каждой строки» — это паралич.
6. Plan mode / предварительный план в инструментах потока: агент сначала показывает план, ты утверждаешь. *[SME-slot: как это включается в инструменте потока, 1 скрин]*
7. Психология ведущего: отклонить лишнее — норма работы, не конфликт с инструментом. Diff — предложение, а не свершившийся факт.

**Мини-практика (квиз, автопроверка):** 5 вопросов: дан план агента из 6 шагов на задачу
«добавить поле в форму» (шаг 3 — «обновлю схему БД», шаг 5 — «заодно отрефакторю валидацию») →
отметить шаги вне scope; выбрать 2 места для чекпоинтов и обосновать выбором из вариантов
(по цене ошибки). Критерий ≥4/5.

**Материалы к производству:** лонгрид, квиз с планом-примером.

**Связи:** ← 2.x (спека = рамка плана), 3.x (контекст = постоянные границы), 1.2 (цена ошибки).
→ 4.2 (дрилл коррекции), 4.3 (живая цикл план→diff→коррекция), W5 (принятый diff идёт на ревью).

**Тиры:** общий; T3 — доп. тезис: prompt chaining / разбиение на сабагентов, чекпоинты на риск
(материал: короткая справка-врезка, не отдельный лонгрид).

## Урок 4.2 — Дрилл «agent drift»: поймать уход от спеки и скорректировать
**Тип:** асинхрон, мини-кейсы 20–25 мин. **4C/ID: PP (part-task practice)** — коррекция агента
на контрольной точке = рутинная операция, которую надо довести до беглости ДО целостной задачи.
Введён по аудиту программы (PP-дефицит W4): в воркшопе петля проживается 1–2 раза, для
автоматизации нужно 6+ повторений — дешевле всего асинхронно.

**Микро-ОР:** студент по фрагменту сессии (спека + план + промежуточный diff) распознаёт тип
дрейфа и выбирает корректное управляющее действие.

**Формат дрилла:** 6 мини-кейсов. В каждом: выдержка из спеки (3–5 строк) + план агента +
промежуточный diff/сообщение агента. Студент: (а) выбирает действие — **принять / отклонить и
переформулировать / остановить и уточнить спеку**; (б) пишет 1–2 предложения «что скажешь агенту»
(сравнение с эталоном-примером, самопроверка).

**Шесть типов дрейфа (по одному на кейс):**
1. **Попутный рефакторинг** — задача про эндпоинт, агент «причесал» соседний модуль. → Отклонить: «верни X, в этой задаче только Y».
2. **Незапрошенная зависимость** — добавил библиотеку, в спеке запрет. → Отклонить + указать на ограничение спеки.
3. **Gold plating** — сделал больше требуемого (кэш, ретраи, конфиг «на будущее»). → Отклонить лишнее: scope из спеки.
4. **Молчаливая смена подхода** — в плане был способ A, в diff способ B без объяснения. → Остановить: «почему B? вернись к плану или обоснуй».
5. **Дыра в спеке вскрылась** — агент спросил/додумал то, чего в спеке нет. → Остановить и уточнить СПЕКУ (не чинить диалогом — правка в артефакт). Ключевой кейс: учит, что коррекция бывает в документ, а не в чат.
6. **Валидный diff** — всё по спеке и плану. → Принять. (Анти-паранойя: учим и принимать; «проверяю каждую строку» — антипаттерн из 1.2.)

Критерий: ≥4/6 действий верно; эталоны формулировок показывают паттерн «факт из diff + ссылка на
пункт спеки/плана + конкретное указание». *[SME-slot: кейсы собрать из реальных сессий на учебном
репо; diff должны быть короткие (10–30 строк), читаемые за минуту]*

**Материалы к производству:** 6 кейсов (спека-фрагмент + план + diff + эталон), разметка в LMS.

**Связи:** ← 4.1 (модель чекпоинтов). → 4.3 (на воркшопе та же петля вживую), 4.4 (ДЗ требует
историю коррекций), 5.2 (дрилл-детект дефектов — продолжение линии «читаю чужой diff»).

**Тиры:** T1 — 6 кейсов с подсказками (тип дрейфа назван); T2 — без подсказок; T3 — кейсы 4–6 +
задание «сформулируй правило в контекст-файл, чтобы дрейф не повторялся» (мостик PP → перенос).

## Урок 4.3 — Live-воркшоп W4 (120 мин)
**4C/ID: LT** (worked example «веду агента вживую» → своя задача по плану).
**План готов:** `../06-lessons/lesson-plan-w4.md`.

**Правка плана под blended:** блок 0:15–0:35 «Концепт: вести, не запускать» → 10 мин активации:
блиц по 1 кейсу дрилла 4.2 («прими/отклони — почему?»). Освободившееся время → worked example
(полная петля план→прогон→diff→коррекция→отклонение лишнего — ядро недели) и обход своей практики
(резать scope, ловить выход за рамки).

**Риск недели (из плана) + контрмера:** эйфория «агент всё может» → обязательный разбор выхода
за scope; после дрилла 4.2 студенты уже видели 6 типов дрейфа — фасилитатор ссылается на них
по именам («это кейс-3, gold plating»). Общий словарь типов дрейфа = germane-выигрыш blended.

**Материалы:** есть (демо agent mode, ветка `w4`, пример прогона с выходом за scope).
Дополнительно: **шаблон agent plan** — см. производство ниже (в `02-materials/` его ещё нет).

## Урок 4.4 — ДЗ: провести задачу агентом по плану → PR
**Тип:** практика проекта, 3 ч. **4C/ID: LT** — шаг 4 сквозного проекта; первая целостная
петля управления на своей задаче.

**Формулировка для студента:**
1. Составь agent plan на текущий шаг проекта (шаги + scope каждого + контрольные точки) по шаблону.
2. Проведи задачу агентом по плану до PR: на каждой контрольной точке сверяй diff со спекой.
3. Зафиксируй историю коррекций: минимум 1 случай «отклонил/переформулировал» с пояснением
   (если агент ни разу не дрейфанул — отметь, на каких чекпоинтах проверял и что именно).
4. PR оформлен: описание + ссылка на спеку (формат из контекст-файла W3).
5. Pre-work W5: прочитай чек-лист ревью `../02-materials/review-checklist-ai-code.md` —
   свой PR из этого ДЗ пойдёт на ревью в W5. **Не вычищай PR вручную до идеала — рабочий
   материал W5 ценнее красоты.**

**Критерий «принято»:** PR соответствует спеке (требования покрыты, ограничения соблюдены);
план с контрольными точками приложен; история коррекций видна. Возврат — с указанием расхождения
diff↔спека.

**Тиры:** T1 — шаблон плана + мелкий scope (1 файл); T2 — свой план, фича или багфикс;
T3 — разбиение на цепочку (chaining/subagents), чекпоинты расставлены по риску, обоснование.

**Материалы:** шаблон agent plan, форма сдачи: PR + план + история коррекций.

## Урок 4.5 — Рефлексия + мост к W5
**Тип:** асинхрон, 10 мин. **4C/ID:** — (метакогнитивный слой).

**Форма рефлексии (Гиббс, вопросы 3 и 5 недельные):**

3. На каком шаге агент ушёл от плана — и какой чекпоинт это поймал (или пропустил)? Какой тип
