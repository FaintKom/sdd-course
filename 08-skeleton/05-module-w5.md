## Урок 5.1 — Таксономия дефектов AI-кода: галлюцинации, лишние правки, тех-долг, security
**Тип:** асинхрон, лонгрид 12–15 мин (текст + скрины diff) + квиз 5 мин. **4C/ID: SI** — «оптика» ревьюера: шесть осей,
по которым AI-код ломается; сам дрилл — в 5.2.

**Микро-ОР:** студент относит дефект AI-кода к одной из шести осей чек-листа и объясняет,
почему AI-код ревьюится иначе, чем человеческий.

**Тезисный контент (порядок подачи):**
1. Хук: diff выглядит идеально — аккуратный, с комментариями, «компилируется». Внутри — вызов несуществующего метода. AI-код **правдоподобен по форме независимо от содержания** — в этом главное отличие от кода человека. *[SME-slot: реальный пример красивого diff с галлюцинацией]*
2. Почему человеческое ревью буксует на AI-коде: объём (агент пишет быстро и много), уверенный тон, отсутствие «запаха неуверенности», по которому мы ловим ошибки джунов.
3. Шесть осей чек-листа (по `review-checklist-ai-code.md`): **A** соответствие спеке · **B** границы и scope · **C** корректность и галлюцинации · **D** качество и тех-долг · **E** безопасность · **F** тесты. По 1 типовому примеру на ось.
4. Опора ревью — **спека рядом с diff**: оси A и B вообще не существуют без спеки (вот зачем были W2–W3). Ревью без спеки = вкусовщина.
5. Связь с типами сбоя (1.1): ось C — сбой «доверие», B — «контекст»/«хаос», D/E/F — «качество». Слайд «4 типа сбоя» — третье возвращение.
6. Порядок прохода: сверху вниз, ✓/✗/n/a; любой ✗ → точечный возврат агенту (не молча чинить руками — принцип курса: управляем агентом, а не переписываем за ним).
7. Blameless-рамка: пойманный дефект = **успех ревью**, а не позор автора. Сейчас тренируемся на чужих дефектах, свой PR — потом (и это нормально).

**Мини-практика (квиз, автопроверка):** 6 коротких описаний дефекта → отнести к оси A–F:
«агент добавил кэширование, которого не просили» (A3-бонус); «переформатировал 3 несвязанных
файла» (B2); «метод `client.fetchAll()` не существует» (C1); «продублировал утилиту из utils/» (D1);
«API-ключ в коде» (E1); «assert подогнан под неправильное значение» (F3). Критерий ≥5/6.

**Материалы к производству:** лонгрид, квиз.

**Связи:** ← 4.4 (PR для ревью готов), 4.5 (предсказания о своём PR), 2.x (спека = опора осей A/B).
→ 5.2 (дрилл), 5.3 (worked example + замер), 6.x (ось F разворачивается в неделю тестирования).

**Тиры:** общий; T2 — вспомни по 1 реальному примеру на любые 3 оси из своей практики;
T3 — каких осей не хватает чек-листу под твой стек (вход в T3-задание воркшопа).

## Урок 5.2 — Дрилл-детект: дефекты в пяти мини-diff
**Тип:** асинхрон, 20–25 мин. **4C/ID: PP (part-task practice)** — тренировка глаза на коротких
diff до воркшопа: на live студент приходит уже «пристрелявшимся», воркшопный замер на наборе
из 10 — вторая серия, а не первая встреча с дефектами.

**Микро-ОР:** студент находит ≥80% подсаженных дефектов в коротких diff, пользуясь чек-листом.

**Формат дрилла:** 5 мини-diff (10–25 строк каждый) + выдержка из спеки к каждому. В каждом diff —
ровно 1 подсаженный дефект T1-уровня (типы из набора: галлюцинация метода; лишние правки вне scope;
заглушка вместо логики; секрет в коде; необработанный пустой ввод). Студент отмечает строку дефекта
и пункт чек-листа, который сработал. Автопроверка; критерий **≥4/5 — тот же порог 80%, что и в ОР
недели** (студент видит свою готовность к воркшопу заранее).

После каждого кейса — эталон: где дефект, какой пункт ловит, **как выглядел бы точечный возврат
агенту** (1 предложение — мостик к коррекциям из 4.2).

*[SME-slot: 5 новых мини-diff — НЕ из воркшопного набора 10 (его не спойлерим, он для замера).
Те же типы дефектов, другой код. Использовать учебный репо-шаблон]*

**Материалы к производству:** 5 мини-diff + спека-фрагменты + эталоны, разметка в LMS.

**Связи:** ← 5.1 (оси), 4.2 (формат «читаю чужой diff» уже знаком). → 5.3 (замер на 10 дефектах),
5.4 (ревью своего PR).

**Тиры:** T1 — с подсказкой «в этом diff дефект оси C»; T2 — без подсказок; T3 — без подсказок +
таймер (беглость: 3–4 мин на diff — реальная скорость ревью в команде).

## Урок 5.3 — Live-воркшоп W5 (120 мин)
**4C/ID: PP + PI** (worked example разбора → практика на 10 подсаженных дефектах с замером →
перенос на свой PR). **План готов:** `../06-lessons/lesson-plan-w5.md`.

**Правка плана под blended:** блок 0:15–0:35 «Концепт: где агент врёт» → 10 мин активации
(оси пройдены в 5.1, глаз пристрелян в 5.2): блиц «какая ось?» по 2 дефектам. Освободившееся
время → расширить PP-практику (блок 1:05–1:35) и разбор своего PR — там germane-пик недели.

**Сохранить обязательно (из плана):** безопасный порядок — сначала подсаженные (чужие) дефекты,
только потом свой PR; замер доли найденного (≥80% — leading indicator курса); фасилитатор первым
показывает свой провальный прогон.

**Материалы:** есть (чек-лист, набор 10 дефектов с раскладкой по тирам, ветка `w5`).

## Урок 5.4 — ДЗ: ревью своего PR по чек-листу + исправление
**Тип:** практика проекта, 2 ч. **4C/ID: LT** — шаг 5 сквозного проекта; перенос PP-навыка
на свой код.

**Формулировка для студента:**
1. Прогони свой PR из W4 по чек-листу (все 6 осей, ✓/✗/n/a). Рядом — спека: оси A/B проверяются только по ней.
2. Сверь находки с предсказаниями из рефлексии 4.5: что угадал, что нет? (1–2 предложения в сдаче)
3. Каждый ✗ — **точечный возврат агенту** на доработку (не правь руками: формулировка по паттерну из 4.2 — факт + пункт спеки + указание). Исправленный PR — итог.
4. Если дефектов не нашлось — задокументируй прохождение по осям (что именно проверил); попроси peer-взгляд на чекпоинте (5.5).

**Критерий «принято»:** чек-лист заполнен по всем осям; найденные дефекты устранены агентом
(видна история возвратов); сверка с предсказаниями есть. Возврат — с указанием оси.

**Тиры:** T1 — по чек-листу как есть; T2 — + secure-coding пункты (ось E расширенно);
T3 — доработай чек-лист под свой стек/команду (+2–3 пункта с обоснованием) и/или сравни свой
детект с автоматическим ревью (Copilot/Codex review): что поймал ты, что — оно, что — никто.

**Материалы:** чек-лист (есть), форма сдачи: заполненный чек-лист + ссылка на исправленный PR + сверка предсказаний.

## Урок 5.5 — Рефлексия + чекпоинт середины курса
**Тип:** асинхрон, 45 мин (15 рефлексия + 30 чекпоинт). **4C/ID:** — (метакогнитивный слой +
формирующая оценка траектории). Материал: `../02-materials/midcourse-checkpoint.md` (готов).

**Форма рефлексии (Гиббс, вопросы 3 и 5 недельные):**
1–2. Стандарт.
3. Какие дефекты ты пропустил (в дрилле/на замере) и почему именно их? Совпали ли предсказания
   4.5 с реальностью твоего PR? (анализ; калибровка самооценки — ядро метакогниции)
4. Стандарт (вывод: «ревью = успех, не позор» — проверяем, что установка села).
5. Какая часть твоего кода не покрыта тестами — и насколько это тревожит? (план → вход в W6)

**Чекпоинт (по `midcourse-checkpoint.md`):** self + peer-review проекта → светофор 🟢/🟡/🔴.
Проверяется: спека/контекст/PR/ревью на месте, scope до финала реалистичен. Ментор подключается
только на 🟡/🔴. Peer-пары назначает фасилитатор (не самовыбор — иначе сильные с сильными).

**Материалы:** форма рефлексии (шаблон курса), чек-лист чекпоинта (есть), механика peer-пар в LMS.
