## Урок 7.1 — Как читать чужой код с агентом: разведка, dependency graph, ADR
**Тип:** асинхрон, лонгрид 12–15 мин (текст + скрины запросов разведки) + квиз 5 мин. **4C/ID: SI** — стратегия эксперта «сначала
понять, потом менять»; агент здесь впервые в роли экскурсовода, а не исполнителя.

**Микро-ОР:** студент проводит разведку незнакомого репо с агентом (структура → зависимости →
точка изменения) до того, как менять хоть строку.

**Тезисный контент (порядок подачи):**
1. Хук: «открыл чужой репо — паника». Нормализация: это у всех, и у сеньоров; разница в том, что у них есть процедура разведки. Сегодня она появится у тебя. *[SME-slot: своя история первого дня в чужом легаси]*
2. Переворот роли: до сих пор агент **писал** — теперь он сначала **объясняет**. Запросы разведки: «объясни, за что отвечает модуль X», «кто вызывает функцию Y», «нарисуй схему зависимостей Z» (Mermaid). Чтение с агентом дешевле чтения в одиночку.
3. Проверка объяснений: агент галлюцинирует и про чужой код (сбой «доверие» — в незнакомом репо ты особенно уязвим, проверить нечем). Правило: ключевые утверждения агента о коде сверяй с самим кодом (открой файл, найди вызов).
4. Что ищем разведкой: точка изменения (где править) → **blast radius** (что зависит от этой точки — что можно задеть) → скрытые контракты (кто полагается на текущее поведение, даже «кривое»).
5. ADR (architecture decision records): если в репо есть — читать до изменений («почему так странно» часто = осознанное решение); своё решение — зафиксировать коротким ADR (T2/T3).
6. Легаси ≠ плохой код: это код, который работает и кормит бизнес. Уважение к работающему — установка недели; «перепишу всё» — самый дорогой сбой («хаос» в масштабе репо).
7. Мостик: вся разведка складывается в спеку и контекст-границы — это 7.2.

**Мини-практика (квиз, автопроверка):** по выданному легаси-репо (клонирован в W6): 5 вопросов
на разведку с агентом — «какой модуль отвечает за X?», «что сломается, если изменить сигнатуру Y?»
(blast radius — выбрать из вариантов), «какое утверждение агента о репо ложно?» (1 подсаженная
галлюцинация — проверить по коду). Критерий ≥4/5. Квиз заставляет реально открыть репо и агента —
это скрытая цель: к воркшопу репо знаком, страх ниже.

**Материалы к производству:** лонгрид, квиз с вопросами по конкретному легаси-репо курса.

**Связи:** ← 6.4 pre-work (репо клонирован), 6.5 (страхи названы). → 7.2 (разведка → спека),
7.3 (worked example), 1.1 (галлюцинации — сбой «доверие», круг замыкается).

**Тиры:** общий; T3 — разведка своего рабочего легаси теми же запросами (параллельно с учебным).

## Урок 7.2 — Спека на изменение в легаси: characterization tests, ограждение scope
**Тип:** асинхрон, лонгрид 12 мин + мини-практика 15 мин. **4C/ID: PI** — чем легаси-спека
отличается от спеки гринфилда; процедура страховки до изменения.

**Микро-ОР:** студент дополняет спеку легаси-специфичными элементами: characterization tests
на текущее поведение и границы, выведенные из blast radius.

**Тезисный контент (порядок подачи):**
1. Спека та же (5 секций W2) — меняются акценты. Не новый инструмент, а перенос: усиливаются секции 3 (границы) и 5 (тесты).
2. **Characterization tests** — главный новый элемент: ДО изменения зафиксировать тестами текущее поведение смежного кода, **даже если оно «кривое»**. Цель — не «правильность», а сигнализация «я что-то задел». Кривое поведение чинится отдельной задачей, не попутно.
3. Секция 3 в легаси — инверсия: в гринфилде перечисляем запретное, в легаси разрешённое («менять ТОЛЬКО `orders/refund.py`; всё остальное репо — запретная зона»). Белый список вместо чёрного.
4. Границы из blast radius (7.1): всё, что зависит от точки изменения, но не входит в задачу → в запретные зоны + на каждое — characterization test.
5. «Не чини попутно» в легаси — закон: агент увидит в старом коде «улучшаемое» повсюду (дрейф типа 1 и 3 из дрилла 4.2 — здесь они особенно соблазнительны). В спеку явно: «код вокруг не трогать, даже если он выглядит плохо».
6. Definition of Done легаси-изменения: тесты новые зелёные + characterization tests зелёные + diff только в белом списке.
7. Порядок работы (процедура недели): разведка → characterization tests → спека с белым списком → план с чекпоинтами → агент → ревью → CI. Это полный цикл W1–W6 — слайд «весь курс в одной задаче» (мотивация: всё было не зря).

**Мини-практика (15 мин, выбор + эталон):** дан фрагмент легаси-кода (из учебного репо) + задача
«исправить расчёт скидки в `pricing.py`». Студент: (а) выбирает из 8 кандидатов 3, которые нужно
покрыть characterization tests (зависимые от точки изменения — по blast radius); (б) из 6 формулировок
границ выбирает корректный белый список; (в) находит в готовой легаси-спеке 1 подсаженную дыру
(«заодно почистить соседний модуль» — нарушение закона «не чини попутно»). Критерий: (а) ≥2/3,
(б) и (в) верно. *[SME-slot: фрагмент и кандидаты — из реального учебного легаси-репо, согласовать
с «минами» воркшопа]*

**Материалы к производству:** лонгрид, мини-практика на материале легаси-репо, слайд «весь курс
в одной задаче».

**Связи:** ← 7.1 (blast radius), 2.x (спека), 4.2 (типы дрейфа). → 7.3 (worked example по этой
процедуре), 7.4 (ДЗ).

**Тиры:** T1 — практика с подсказками; T2 — без; T3 — составь белый список для изменения в своём
рабочем легаси.

## Урок 7.4 — ДЗ: PR в легаси-репо через полный цикл
**Тип:** практика проекта, 3–4 ч. **4C/ID: LT** — кульминация переноса; пиковая нагрузка курса.

**Формулировка для студента:**
1. Доведи изменение, начатое на воркшопе (или возьми задачу из списка недели), до PR в легаси-репо — **полным циклом**: разведка → characterization tests → спека (белый список границ) → план с чекпоинтами → агент → ревью по чек-листу → зелёный CI.
2. В PR-описании — ссылки на все артефакты цикла (спека, план, чек-лист) + 3–5 строк: что было самым трудным и какой инструмент курса спас.
3. Критично: **смежное цело** = characterization tests зелёные; **scope соблюдён** = diff только в белом списке.

**Критерий «принято»:** тесты (новые + characterization) зелёные; diff в белом списке; артефакты
цикла приложены. Возврат — с указанием, что именно вышло за рамки.

**Поддержка на яме #2 (организационно):** дедлайн мягче обычного (+2 дня против стандартного);
канал недели мониторится фасилитатором активнее; пары с воркшопа продолжают страховать друг друга
(peer-пинг в середине недели — механика в LMS); застрял >40 мин — пишешь в канал, это норма (W1).

**Тиры:** T1 — локализованный багфикс из выданного списка (1 файл белого списка); T2 — фича или
рефакторинг в учебном легаси; T3 — изменение в своём рабочем легаси + бюджет токенов как
ограничение + короткий ADR решения.

**Материалы:** список задач недели по тирам (под «мины» репо), форма сдачи: PR + артефакты.
*[SME-slot: список задач — 3–4 на тир, каждая проверена прогоном: решаема за 2–3 ч с инструментами курса]*
