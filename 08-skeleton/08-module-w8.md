## Урок 8.1 — Как доказать управление: рубрика, структура демо
**Тип:** асинхрон, лонгрид 12 мин + черновик плана рассказа 5 мин. **4C/ID: SI** —
модель «что такое доказанное управление»; прозрачность суммативной оценки (рубрика известна
заранее — снимает половину тревоги защиты).

**Микро-ОР:** студент раскладывает свой проект по 6 критериям рубрики и составляет план
5–7-минутного рассказа, в котором показывает управление, а не код.

**Тезисный контент (порядок подачи):**
1. Рамка: защита = **защита, не экзамен**. Оценка дуальная (принято / доработка), уровень T1–T3 поверх. Главная ошибка защиты — «вот код, он работает»: код не доказывает управление.
2. Что доказывает управление: артефакты процесса. Спека ограничила агента (видно по diff) → контекст удержал границы → план с чекпоинтами → отклонённые лишние правки (история коррекций!) → чек-лист ревью с пойманными дефектами → тесты и зелёный CI.
3. Рубрика по 6 критериям (из `defense-rubric.md`), на каждый — «что показать на экране»: 1-спека (файл + ссылка из PR), 2-контекст (AGENTS.md), 3-управление (план + коррекции), 4-ревью (заполненный чек-лист), 5-тесты (CI), 6-демонстрация (сам рассказ по этапам).
4. Структура рассказа 5–7 мин: задача (30 сек) → цикл по этапам с артефактами (4–5 мин, по 30–60 сек на этап) → где ИИ сбоил и как удержал рамками (1 мин — это сильнейшая часть, не прячь сбои!) → рефлексия Гиббса (2–3 мин, входит в защиту).
5. Чек-лист «что проверяющий смотрит в репозитории» (из рубрики) — прогнать ДО защиты, это и есть план финализации (8.4).
6. Типичные вопросы эксперта (подготовься): «почему агент не вышел за scope?», «покажи отклонённую правку», «что бы сделал иначе?». *[SME-slot: 5–7 реальных вопросов, которые задаст эксперт]*

**Мини-практика (5 мин, без оценки):** черновик плана рассказа — 6 строк (по критерию на строку:
каким артефактом докажу). Сохраняется в LMS, дорабатывается в 8.4.

**Материалы к производству:** лонгрид, шаблон плана рассказа (6 строк).

**Связи:** ← 7.5 pre-work (рубрика прочитана), весь курс (артефакты W1–W7). → 8.2 (мини-цикл
как репетиция беглости), 8.4 (финализация по чек-листу), 8.3 (защита).

**Тиры:** общий; T3 — в план рассказа добавить «обоснование решений на развилках» и
«ускорение vs тех-долг» (критерии «Отлично» рубрики).

## Урок 8.2 — Цикл за 60 минут: незнакомая задача полным циклом
**Тип:** асинхрон, таймбокс 60 мин. **4C/ID: LT — второй целостный проход** (новый класс задач:
та же компетенция, незнакомый материал, сжатое время, без опоры). Смысл: сквозной проект рос
8 недель с поддержкой — это один проход. Перенос (transfer, цель программы) доказывается вторым
проходом на новом материале. Заодно снимает главное возражение против spec-driven: «в реальной
работе нет времени на спеки» — есть, если цикл занимает 15 минут подготовки.

**Микро-ОР:** студент проходит полный spec-driven цикл на незнакомой мини-задаче за 60 минут
без шаблонов-подсказок (T2+).

**Формат (инструкция студенту):**
Выдаётся мини-задача (новая, НЕ из твоего проекта: мелкая фича/багфикс в учебном репо — модуль,
которого ты не касался). Таймбокс 60 минут, рекомендованная раскладка:
- 0:00–0:15 — **spec-lite** (компресс-шаблон 1 стр.: цель, 3 требования с AC, границы, DoD, 2 теста);
- 0:15–0:20 — границы: проверь, что контекст-файл покрывает зону задачи;
- 0:20–0:25 — план агента (3–5 шагов, 1–2 чекпоинта);
- 0:25–0:40 — прогон с коррекцией на чекпоинтах;
- 0:40–0:55 — ревью по чек-листу (сокращённый прогон) + тесты;
- 0:55–1:00 — итог: критерии DoD да/нет.

Сдача: артефакты мини-цикла (spec-lite, план, чек-лист, diff/PR) + 2 строки «что выкинул из
полного цикла и почему это было безопасно». Формативная оценка (не влияет на защиту): самопроверка
по чек-листу + блиц-разбор 2–3 работ на воркшопе.

**Ключевой учебный эффект:** студент сам видит, какие элементы цикла сжимаются (спека → 1 стр.),
а какие не выкидываются никогда (границы, чекпоинт, ревью). Это и есть интериоризованная схема,
а не следование шаблону.

**Тиры:** T1 — задача из списка + spec-lite, заполненный наполовину (последняя опора курса);
T2 — задача из списка, без заготовок; T3 — таймбокс 40 мин ИЛИ реальная мелкая задача из своей
работы (закрытие: «в понедельник так и делай»).

**Материалы к производству:** 3–5 мини-задач (прогнаны: решаемы за 25–35 мин чистого времени),
**шаблон spec-lite (1 стр.)** — добавить в `02-materials/` (пригодится выпускникам как рабочий
инструмент), инструкция таймбокса. *[SME-slot: мини-задачи; spec-lite — сжать spec-template.md
до 1 страницы, сохранив 5 секций]*

**Связи:** ← весь цикл W2–W6 (применяется целиком). → 8.3 (блиц-разбор + материал для ОС эксперта:
второй проход = доказательство переноса), 8.5 (план переноса опирается на этот опыт).

## Урок 8.4 — ДЗ: финализация репозитория
**Тип:** практика проекта, 4 ч (выполняется ДО воркшопа 8.3 — единственная неделя, где ДЗ
предшествует live). **4C/ID: LT** — сборка и артикуляция.

**Формулировка для студента:**
1. Прогони репозиторий по чек-листу «что проверяющий смотрит» (из рубрики): спека в репо + ссылка
   из PR; AGENTS.md с границами; история PR с коррекциями; заполненный чек-лист ревью; тесты +
   зелёный CI; README (что делает, как запустить, как проверялось).
2. README — единственный новый артефакт: 0.5–1 стр., карта цикла со ссылками на артефакты
   («спека → [link], план → [link]…»). Не пиши прозу — сделай навигацию для эксперта.
3. Доработай план рассказа из 8.1 → прогони рассказ 1 раз с таймером (5–7 мин жёстко; запись
   на телефон — лучший самоконтроль). Опц.: прогон с peer-парой из W7.
4. Подготовь финальную рефлексию Гиббса (2–3 мин, 5 пунктов из рубрики) — она часть защиты.

**Критерий «принято»:** чек-лист рубрики — все 6 пунктов на месте; рассказ уложен в таймбокс.
(Формальная приёмка — сама защита 8.3.)

**Тиры:** T1 — по чек-листу; T2 — + в README блок «эффект: что поймал контролем» (метрики:
дефекты, отклонённые правки); T3 — + обоснование решений на развилках, рефлексия «ускорение vs
тех-долг» (критерии «Отлично»).

**Материалы:** чек-лист (есть в рубрике), шаблон README-навигации (5 строк), форма записи на слоты.

## Урок 8.5 — Рефлексия курса + план переноса
**Тип:** асинхрон, 20 мин, после защиты. **4C/ID:** — (закрытие метакогнитивного слоя и
аффективной дуги курса; «новый AE» цикла Колба — обучение продолжается за пределами курса).

**Форма (3 части):**
1. **Пересмотр карты доверия/контроля из W1** (главный измеримый сдвиг установки): открой карту
   из 1.4 → что переместил бы сейчас между зонами и почему? 3–5 перемещений с аргументом
   «научился проверять X → могу доверить Y». Это закрытие арки: в 1.2 обещали, что границы
   сместятся с ростом навыка контроля.
2. **План переноса** (из заготовки 7.5 + опыта 8.2): первая рабочая задача, к которой применишь
   цикл; что из инструментов возьмёшь как есть (spec-lite? чек-лист?); что адаптируешь под
   команду; первый шаг в понедельник.
3. **Обратная связь курсу** (для команды, не оценивается): где было труднее всего / что дало
   больше всего / NPS. + указатели «куда дальше» (темы v2 из канона §7: security AI-разработки,
   локальные модели, governance команды).

**Материалы:** финальная форма рефлексии (отдельная, не недельный шаблон), указатели v2.
