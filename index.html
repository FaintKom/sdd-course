program-and-outcomes.md

Программа и образовательные результаты — «Управляемая разработка с ИИ» (SDD)

Главный документ курса. Канон программы — вкладка «ТЗ на ленд» (8 недель воркшопов).
Результаты — в формате ABCD (Audience · Behavior · Condition · Degree), измеримые.
Замаплены на реальные рабочие задачи специалиста (вкладка «Слепок рынка»).
4C/ID: LT — learning task, SI — supportive info, PI — procedural info, PP — part-task practice.

0. Идеология и большая идея

Идеология (ползунок Широ): ведущая — Social Efficiency (курс от запроса рынка: ОР =
трудоустраиваемые поведения, аутентичные задачи, бенчмарк конкурентов); опора — Learner-Centered
(тиры T1–T3, гейт по способности, свой проект / стек / тема). Это критерий спорных решений: при
конфликте выбираем рыночную пользу + автономию студента, а не академический канон.

Пример применения: «давать ли готовый репо-шаблон?» → Social Efficiency = да (экономим время на
реальную работу), а не «собери с нуля ради чистоты».

Большая идея (enduring understanding):

Ты управляешь ИИ через спецификацию и контроль качества — не угадываешь промптами.

Цель на перенос (transfer): студент сам применяет spec-driven цикл к новой задаче / стеку /
инструменту — без инструкции. Это главный признак, что научили, а не натаскали.

Сквозные ключевые вопросы (вешаем в W1, возвращаемся на чекпоинте и защите):
- Как заставить ИИ сделать ровно то, что нужно — и доказать, что он это сделал?
- Где ИИ доверять, а где контролировать?

1. Программа (канон, 8 недель)

Формат: живые воркшопы, один поток. Повторяете за экспертом + параллельно собираете
сквозной проект (реальная/учебная задача: фича / багфикс / рефакторинг). Каждая неделя =
шаг проекта. Финал — репозиторий с полным циклом + защита перед экспертом.

Нед. Тема Что осваивает Артефакт недели
W1 Spec-Driven Development: почему промпты ломаются модель сбоя агента (контекст/доверие/хаос/качество); ментальная модель «менеджер отделов» (что отдать ИИ, что проверять); зачем spec-driven карта зон доверия/контроля + выбор темы проекта
W2 От задачи к спецификации формализация задачи в спеку (требования, ограничения, AC, DoD, тесты) спецификация (5 секций)
W3 Управление контекстом проекта контекст-файл для агента (архитектура, конвенции, границы) AGENTS.md / CLAUDE.md
W4 Постановка задач AI-агенту план работы агента, scope, контрольные точки agent plan + PR по спеке
W5 Контроль качества AI-кода ревью AI-кода по чек-листу, ловля типовых дефектов заполненный чек-лист ревью + исправленный PR
W6 Тестирование AI-разработки тест-стратегия, негативы, coverage как критерий тесты + coverage-отчёт, зелёный CI
W7 Работа с легаси и существующим кодом изменение через спеку в чужом коде без сноса смежного PR в легаси-репо
W8 Сквозной проект и защита сборка полного цикла + демонстрация управления ИИ репозиторий + защита

Полный цикл (Экран 5 ленда): задача → спецификация → контекст → реализация → тесты → ревью → доработка → защита.

2. Образовательные результаты программы (terminal objectives)

Что выпускник умеет делать после курса. Каждый — ABCD + компонент 4C/ID + связь с рабочей задачей.

ОР-1 — Полный spec-driven цикл [LT]
- A выпускник · B проводит задачу (фича/багфикс/рефактор) через полный spec-driven цикл, управляя AI-агентом · C в репозитории, на защите перед экспертом · D доходит до прохождения всех критериев готовности без ручного переписывания сгенерённого кода.
- ← Слепок рынка: интеграция всех задач junior+middle.

ОР-2 — Формализация задачи в спецификацию [PI]
- A выпускник · B превращает задачу в спецификацию (требования, ограничения, acceptance criteria, DoD, тесты) · C для реальной/учебной задачи по шаблону · D спека покрывает все 5 секций и однозначно ограничивает агента.
- ← junior task 1 (первичная спека), middle task 1 (полная спека).

ОР-3 — Управление контекстом [PI]
- A выпускник · B готовит контекст проекта (архитектура, конвенции, границы изменений) · C через AGENTS.md/CLAUDE.md в репозитории · D агент выполняет задачу не выходя за границы и не трогая запрещённые файлы.
- ← junior task 2 (контекст для агента), middle task 2/3.

ОР-4 — Постановка и ведение агента [LT]
- A выпускник · B ставит задачу агенту в scope и ведёт по плану с контрольными точками · C на ограниченном scope в репозитории · D получает diff/PR, соответствующий спеке; лишние изменения отклонены.
- ← junior task 3, middle task 2.

ОР-5 — Контроль качества AI-кода [PP]
- A выпускник · B проверяет AI-generated code тестами и чек-листом ревью · C имея diff и спеку · D выявляет ≥80% типовых дефектов (галлюцинации, лишние правки, тех-долг) до мерджа.
- ← junior task 4, middle task 4/5.

ОР-6 — Применение в легаси и встраивание в процесс [LT]
- A выпускник · B безопасно применяет ИИ в существующем/коммерческом коде и встраивает spec-driven в личный/командный workflow · C в легаси-репозитории, не им написанном · D изменение проходит тесты, не ломает смежное, ограничено scope; процесс повторяем и не завязан на конкретный инструмент.
- ← middle task 3 (управление в существующей базе).

Бенчмарк: как другие учат Spec-Driven Development и «AI-агенты для разработчиков»

Основа для принятия решений по дизайну курса. Веб-ресёрч инструментов и курсов SDD.
Источники — внизу.

TL;DR

Рынок сошёлся на едином backbone SDD: Constitution/Steering → Specify → (Clarify) → Plan →
Tasks → Implement → Verify. Инструменты (GitHub Spec Kit, AWS Kiro, Tessl) учат механике;
курсы (DeepLearning.AI, Udemy) — смене мышления (vibe-coding → инженерия). Почти всё —
короткое, self-paced, видео+квиз, один follow-along проект. Незакрытые пробелы рынка:
brownfield/легаси, governance на команду, drift detection, и graded capstone с реальным ревью.

Backbone (де-факто стандарт)

  1. Governance первым — Constitution (Spec Kit) / Steering files (Kiro) / CLAUDE.md+AGENTS.md (Udemy).
  2. Specify — что/зачем, без реализации; user stories + acceptance criteria; измеримый успех.
  3. Clarify — явный гейт устранения неоднозначности (NEEDS CLARIFICATION) ДО планирования.
  4. Plan — архитектура/решения под существующий код + конституцию.
  5. Tasks — атомарные, ревьюируемые, с зависимостями (пометка параллельных).
  6. Implement — агент выполняет, human-in-the-loop, контроль дрейфа.
  7. Verify — тесты-как-спека, кросс-артефактный анализ, реплан между фичами.
    Сквозное: спеки в git; тесты из спеки, не после кода; EARS — эмерджентный стандарт AC;
    агенты заменяемы/agnostic; человек аппрувит на гейтах.

Собрать самодостаточный HTML-разбор курса как LXD: как студенты работают, что готовить

LXD-линзы (подняты как скиллы)

  • ADDIE (Branch 2009; Dick&Carey 2014) — 5 фаз, сквозной alignment objective→assessment→content.

    worked-example fading, expertise reversal.
  • Action Mapping (Moore 2017) — поведение-first, goal→actions→filter (skill vs env/motiv),
    практика > контент.

  • Канон программы (вкладка «ТЗ на ленд», Экран 8): 8 недель, живые воркшопы, старт июль.

Метрики качества (PASS/FAIL, итерирую пока не ≥9/10)

  1. Точность источников — каждый факт прослеживается к таблице/опросу; конфликты разрешены явно; ничего не выдумано.

  3. Когнитивная нагрузка — каждый блок разобран intrinsic/extraneous/germane + тактики; риск «setup съел воркшоп» закрыт явно.
  4. Ясность работы студента — на каждом этапе: что делает студент, pre-work, подготовка фасилитатора, цель, результат, артефакт; почасовой run-of-show.
  5. Дифференциация ЦА — гейт по участию (не по «вкусу»); тиры T1–T3; expertise reversal учтён; junior+→senior обслужены.
  6. Action-mapping — поведение-first; goal→actions→filter (skill vs env/motiv); практика>контент; env-гэпы (setup, платные токены) вынесены в не-тренинговые фиксы.
  7. Артефакты — конкретные деливераблы на каждой неделе, замаплены на портфолио/резюме и сквозной проект.
  8. Самодостаточный HTML — один файл, рендерится офлайн, sticky-TOC, печатается, профессионально.
  9. Оценка — Kirkpatrick L1–L4 + leading indicators, замаплено на реальные метрики групп (COR, NPS, доходимость, % целей).
  10. Actionability — по доку команда может провести курс: run-of-show, чек-листы фасилитатора, риски+митигации, материалы-чеклист, прототип.

Цель: ≥9/10 PASS, ни одной критической FAIL.

Шаблон спецификации задачи (Spec)

Артефакт недели W2 «От задачи к спецификации». Студент заполняет на свою задачу
сквозного проекта. Спека — это ТЗ для AI-агента: она удерживает его в рамках.
Критерий готовности спеки: заполнены все 5 секций; по ней можно проверить результат.

0. Метаданные

  • Задача / тикет: <ссылка или ID>
  • Тип: фича / багфикс / рефакторинг
  • Автор спеки: <имя>
  • Целевой репозиторий + ветка:

1. Цель и контекст (зачем)

Одно-два предложения: какую пользователю/бизнесу проблему закрываем. Не «как», а «зачем».

Пример: «Пользователь не может отменить заказ после оплаты — растёт нагрузка на поддержку.
Нужна самостоятельная отмена в окне 30 минут.»

2. Требования (что должно быть сделано)

Список наблюдаемого поведения. Каждый пункт проверяем.
- [ ] <требование 1>
- [ ] <требование 2>

User story (опц., для middle+):
Как <роль>, я хочу <действие>, чтобы <ценность>.

Acceptance criteria в нотации EARS (стандарт индустрии, тестируемо — см. бенчмарк):
- WHEN <событие> THE SYSTEM SHALL <поведение>
- IF <условие>, THEN THE SYSTEM SHALL <поведение>
- WHILE <состояние> THE SYSTEM SHALL <поведение>

Пример: WHEN пользователь жмёт «отменить» в окне 30 мин THE SYSTEM SHALL отменить заказ и вернуть оплату.

Clarification gate (до передачи агенту)

Помечай всё неясное [NEEDS CLARIFICATION: вопрос] и разрешай ДО планирования/реализации.
Спека с незакрытыми NEEDS CLARIFICATION в работу не идёт.

3. Ограничения и границы (чего агент НЕ делает)

Самая важная секция против «агент сносит рабочее решение». Явно:
- Не менять файлы: <пути / модули>
- Не трогать: <публичные API, схему БД, конфиги прод>
- Технические ограничения: <язык, версии, паттерны, без новых зависимостей и т.п.>
- Scope: только . Всё за рамками — отдельной задачей.

4. Критерии готовности (Definition of Done)

Бинарно проверяемо. «Принято», когда выполнены ВСЕ:
- [ ] Поведение из секции 2 реализовано
- [ ] Тесты из секции 5 проходят (CI зелёный)
- [ ] Ограничения секции 3 соблюдены (diff не выходит за scope)
- [ ] Ревью по чек-листу пройдено (см. review-checklist-ai-code.md)
- [ ] PR оформлен: описание + ссылка на эту спеку

5. Тесты / приёмочные сценарии

Как проверим. Прямой happy-path + негативы + edge.

Сценарий 1 (happy): дано <...> когда <...> тогда <...>
Сценарий 2 (негатив): дано <...> когда <...> тогда <ошибка/отказ>
Сценарий 3 (edge): <...>

Для middle+/T3 — формат Given-When-Then (Gherkin).

Чек-лист самопроверки спеки (перед передачей агенту)

  • [ ] Цель отвечает на «зачем», не на «как»
  • [ ] Каждое требование наблюдаемо и проверяемо
  • [ ] Есть явный список запретных файлов/зон
  • [ ] DoD бинарный (можно сказать да/нет по каждому пункту)
  • [ ] Есть хотя бы 1 негативный и 1 edge-сценарий
  • [ ] По спеке третий человек поймёт, что и в каких рамках делать

Тиры (W2)

  • T1 (junior+): заполнить по worked example, happy-path фича, секции 1–4.
  • T2 (middle): своя задача, все 5 секций, прогнать агента с/без спеки и зафиксировать разницу.
  • T3 (senior/lead): Gherkin + ADR-ссылка, спека на легаси-кейс, годная как командный стандарт.

Учебное упражнение «с/без спеки»

  1. Прогнать агента на задаче БЕЗ спеки → сохранить diff_0.
  2. Прогнать ту же задачу СО спекой → diff_1.
  3. Зафиксировать разницу: лишние файлы, выход за scope, недостающее поведение.
    Это germane-нагрузка: строит схему «спека управляет агентом».

Чек-лист ревью AI-сгенерированного кода

Артефакт недели W5 «Контроль качества AI-кода». Студент прогоняет diff агента по
этому чек-листу. Цель недели: поймать ≥80% типовых дефектов ДО мерджа.
Тренируемся на наборе с подсаженными дефектами (см. вторую половину файла).

Как пользоваться

  1. Открыть diff/PR агента рядом со спекой (spec-template.md).
  2. Пройти по пунктам сверху вниз, отметить ✓ / ✗ / n/a.
  3. Любой ✗ → возврат агенту с точечным запросом на доработку.
  4. Зафиксировать, какие дефекты нашли (для метрики ревью-детект).

A. Соответствие спеке

  • [ ] Реализовано всё из секции «Требования» спеки
  • [ ] Поведение совпадает с приёмочными сценариями (секция 5 спеки)
  • [ ] Нет «бонусной» функциональности, которой не просили

B. Границы и scope (против «агент трогает лишнее»)

  • [ ] Изменены только файлы в рамках scope; запретные зоны не тронуты
  • [ ] Нет случайных правок форматирования/импортов в несвязанных файлах
  • [ ] Публичные API / схема БД / конфиги не изменены без явного требования
  • [ ] Нет новых зависимостей, если спека их не разрешала

C. Корректность и галлюцинации

  • [ ] Вызываемые методы/поля/библиотеки реально существуют (не выдуманы агентом)
  • [ ] Граничные случаи и ошибки обрабатываются, не только happy-path
  • [ ] Нет «заглушек», выдаваемых за реализацию (TODO, pass, return null/mock)
  • [ ] Логика соответствует, а не просто «компилируется/запускается»

D. Качество и тех-долг

  • [ ] Нет дублирования уже существующего в проекте кода
  • [ ] Именование и стиль соответствуют конвенциям репо
  • [ ] Сложность оправдана задачей (агент не перемудрил)
  • [ ] Читаемо для следующего разработчика без расшифровки

E. Безопасность (база)

  • [ ] Нет секретов/токенов/ключей в коде или логах
  • [ ] Ввод валидируется; нет очевидных инъекций
  • [ ] Нет небезопасных вызовов (eval, shell с конкатенацией, отключённый TLS)

F. Тесты (мост к W6)

  • [ ] Критичная логика покрыта тестами
  • [ ] Есть негативные сценарии, не только happy-path
  • [ ] Тесты реально проверяют поведение, а не подогнаны под результат

Вердикт

Промежуточная проверка проекта (checkpoint, конец W5 / начало W6)

Две цели: (1) скорректировать траекторию студента до финала; (2) сократить стоимость ревью —
триаж: ментор тратит время только на проблемных, остальное закрывают self- и peer-review.
Ставится после W5 (уже есть спека, контекст, ≥1 проведённый агентом PR, ревью-чек-лист) и ДО W7,
чтобы студент успел исправиться.

Что проверяем (накоплено за W2–W5)

  1. Тема проекта реалистична по объёму к W8 — не раздута и не пуста.
  2. Спека (5 секций, EARS) есть и однозначна; нет открытых NEEDS CLARIFICATION.
  3. Контекст-файл (AGENTS.md/CLAUDE.md) реально удерживает агента — не трогает запретное.
  4. ≥1 задача проведена агентом в scope, PR соответствует спеке (лишнее отклонено).
  5. Ревью-чек-лист применён хотя бы раз (есть заполненный).

Формат (дёшево-сначала)

  1. Self-check по чек-листу — студент сам, ~10 мин.
  2. Peer-review в парах — взаимно по тому же чек-листу, ~20 мин. Обратная связь без ментора.
  3. Светофор по итогу: 🟢 ок / 🟡 риск / 🔴 off-track.
  4. Ментор разбирает только 🟡/🔴 (~15 мин/чел). 🟢 — авто-ок + короткий письменный комментарий.

Критерии светофора

  • 🟢 ок — все 5 пунктов выполнены.
  • 🟡 риск — 1–2 пункта частично; поправимо за неделю.
  • 🔴 off-track — спека пустая / тема раздута / агент без границ / нет ни одного проведённого PR.

Корректировка траектории

  • Студенту 🟡/🔴: один конкретный план исправления (сузить scope / переписать спеку / добить контекст) — до W7.
  • Команде: агрегат светофоров по группе. Если у многих 🔴 в одном пункте (напр. контекст) — фасилитатор
    добавляет разбор этого места в W6/W7. Чекпоинт = сигнал, куда подкрутить курс.

Экономия ревью (деньги)

  • Триаж. Ментор работает только с 🟡/🔴. При доле 🟢 ~50% → вдвое меньше менторо-часов на этом этапе.
  • Ранняя ловля off-track. Дешевле исправить раздутый scope на W5–6, чем переписывать на защите.
    Меньше крупных провалов на W8 → короче защита → меньше времени эксперта.
  • Peer-review переносит часть нагрузки на студентов и сам тренирует навык ревью (germane load).
  • Наследование критериев. Рубрика защиты W8 использует те же критерии → на защите проверяем дельту,
    а не с нуля. Ещё экономия времени эксперта.

Таксономия Блума для курса «Управляемая разработка с ИИ»

Подробная разметка образовательных результатов по таксономии Блума (ред. Андерсон–Кратволь).
Цель: проверить, что курс ведёт студента вверх по уровням и не застревает на «понять».
Таксономия двумерная: уровень мышления × тип знания.

Двумерная сетка

Уровни мышления (снизу вверх):
1. Запомнить — воспроизвести факты, термины.
2. Понять — объяснить смысл, привести пример, классифицировать.
3. Применить — использовать процедуру в новой ситуации.
4. Проанализировать — разложить на части, найти связи, отличить.
5. Оценить — судить по критериям, проверить, раскритиковать.
6. Создать — собрать новое целое, спроектировать.

Типы знания:
- Фактический — термины, факты (что такое спека, EARS).
- Концептуальный — модели, принципы (почему промпты ломаются, 4 типа сбоя).
- Процедурный — как делать (написать спеку, провести агента, ревью).
- Метакогнитивный — управление своим мышлением (где доверять ИИ, что проверять, рефлексия).

03-lesson-01/lesson-plan-w1.md

План урока W1 — Spec-Driven Development: почему промпты ломаются

Первый воркшоп. Тип: SI (supportive info) + старт сквозного проекта. Длительность ≈2 ч, live.
Задача урока — не научить кодить с ИИ, а переформатировать мышление: проблема не в том,
что ИИ плохо пишет код, а в том, что задача плохо формализована. + завести каждого в проект.

Образовательный результат (ABCD)

  • A студент · B объясняет, почему хаотичные промпты ломаются, и классифицирует сбой агента
    (контекст / доверие / хаос / качество) · C на 4 реальных кейсах диалога с ИИ ·
    D верно атрибутирует причину сбоя в ≥3 из 4.

Цель для преподавателя

  • Показать разрыв «промптинг vs управление»; дать рамку 4 типов сбоя.
  • Снять страх/завышенные ожидания, задать тон «инженерный процесс, не вайбкодинг».
  • Запустить environment doctor и помочь каждому выбрать тему сквозного проекта.

Нормы группы — психобезопасность (задать на W1, держать весь курс)

Курс про показ чужих и своих ошибок ИИ-кода (ревью, разбор сбоев, защита). Без безопасной
среды студенты прячут факапы → ломается ядро (контроль качества). Проговорить в начале W1:
- Blameless: ошибка агента и твоя ошибка в управлении им — учебный материал, не позор.
- «Плохой» diff, пойманный дефект, откат — это успех ревью, а не провал.
- На разборах критикуем спеку/процесс, не человека.
- Спросить «я застрял» — норма; именно так управляют реальной командой.
Фасилитатор моделирует сам: показывает собственный провальный прогон агента первым.

Что студент уносит (артефакты)

  • Заполненная карта зон доверия/контроля (что отдаю агенту / что проверяю).
  • Выбранная тема сквозного проекта (фича / багфикс / рефактор).

Предусловия (pre-work студента)

См. ../02-materials/pre-work-environment-checklist.md: поднятое окружение + кандидат на проект.

Run-of-show (минута-в-минуту, ≈120 мин)

Время Блок Что делает преподаватель Что делают студенты Нагрузка
0:00–0:10 Вступление карта курса (модуль = шаг проекта), правила, тон знакомятся, задают вопросы —

Проверка достижения результата

  • На кейсах 3–4 студент сам называет тип сбоя → ≥3 из 4 верно.
  • Карта доверия/контроля заполнена связно.
  • Тема проекта валидна (это фича/багфикс/рефактор, а не «сделать приложение с нуля без кода»).

ДЗ к W2

  • Дозаполнить карту доверия/контроля (если не успел).
  • Прочитать примеры хорошей/плохой спеки (выдаются), описать свою задачу 3–5 предложениями.
  • Критерий «принято»: тема валидна, карта заполнена. ~1–2 ч.

Тиры

  • T1 (junior+): кейсы и шаблон карты даны; тема — из учебного списка.
  • T2 (middle): приносит свой кейс сбоя из работы; тема — своя рабочая задача.
  • T3 (senior/lead): формулирует командную эвристику «где ИИ применять, где нет»; тема — легаси.

Риски урока и митигация

  • Setup съедает время → большая часть в pre-work; doctor строго 15 мин; снапшот-ветка как запас.
  • Расфокус на инструменты («а Cursor или Claude?») → один основной инструмент потока, остальное опционально, не спорим на уроке.
  • Завышенные ожидания («ИИ всё сам») → прямо проговорить анти-портрет и рамку курса.

03-lesson-01/dialogue-cases-w1.md

W1 — 4 кейса-диалога с AI-агентом (+ answer key)

Учебный материал W1. Студенты атрибутируют тип сбоя. Рамка из 4 типов (Экран 3 ленда):
контекст · доверие · хаос · качество. Кейсы 1–2 разбирает эксперт (worked example),
3–4 — студенты сами. Цель: ≥3 из 4 верно.

Рамка: 4 типа сбоя

Тип Суть Симптом
контекст агент не знает проект, подгоняет под своё решение трогает лишние файлы, игнорит архитектуру
доверие ИИ галлюцинирует, код «вроде работает» выдуманные методы, незаметные баги
хаос всё на промптах наугад, нет системы каждую задачу с нуля, не воспроизводимо
качество запускается, но плохо тех-долг, лишние правки, не ревьюится

Кейс 1 (worked example, провальный)

Промпт: «Добавь отмену заказа.»
Что сделал агент: добавил эндпойнт + переписал модель Order, заодно поменял формат даты во всём проекте, тронул 9 файлов.
Симптом: изменения далеко за рамками задачи, сломал смежное.

Answer key: контекст (не знал границ проекта, не было scope) — частично качество.

Кейс 2 (worked example, рабочий)

Промпт: спека: «Отмена заказа в окне 30 мин. Не менять: схему БД, формат даты. Только order_service.py. AC: отмена доступна <30мин; >30мин → 409. Тест на оба.»
Что сделал агент: изменил 1 файл, добавил 2 теста, уложился в AC.

Answer key: сбоя нет — спека дала контекст, границы, критерии. Эталон.

Кейс 3 (студенты атрибутируют)

Промпт: «Напиши функцию выгрузки отчёта через библиотеку reportlib.»
Что сделал агент: сгенерил код с вызовом reportlib.export_pdf(). Метода нет в библиотеке — выдумал.

Answer key: доверие (галлюцинация API). Ловится тестом/проверкой существования метода.

Кейс 4 (студенты атрибутируют)

Промпт: «Почини баг с дублями в списке.» (без тикета, без шагов воспроизведения)
Что сделал агент: добавил set() поверх результата — дубли ушли, но сломалась сортировка; работает «на глаз».

Answer key: качество (+ хаос: задача не формализована, нет критерия готовности). Решил симптом, создал регресс.

Как вести разбор

  1. Показать промпт + результат.
  2. Спросить: «какой тип сбоя? почему?»
  3. Связать с лечением на курсе: контекст→W3, доверие→W5/W6, хаос→W2, качество→W5.
    Это закладывает карту курса в голову с первого урока.
