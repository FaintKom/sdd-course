# Программа и образовательные результаты — «Управляемая разработка с ИИ» (SDD)

> Главный документ курса. Канон программы — вкладка «ТЗ на ленд» (8 недель воркшопов).
> Результаты — в формате **ABCD** (Audience · Behavior · Condition · Degree), измеримые.
> Сопоставлены с реальными рабочими задачами специалиста (вкладка «Слепок рынка»).
> 4C/ID: LT — learning task, SI — supportive info, PI — procedural info, PP — part-task practice.

---

## 0. Идеология и большая идея

**Идеология (ползунок Широ):** ведущая — **Social Efficiency** (курс от запроса рынка: ОР =
трудоустраиваемые поведения, аутентичные задачи, бенчмарк конкурентов); опора — **Learner-Centered**
(тиры T1–T3, отбор по способности, свой проект / стек / тема). Это критерий спорных решений: при
конфликте выбираем рыночную пользу + автономию студента, а не академический канон.
> Пример применения: «давать ли готовый репо-шаблон?» → Social Efficiency = да (экономим время на
> реальную работу), а не «собери с нуля ради чистоты».

**Большая идея (enduring understanding):**
> Ты управляешь ИИ через спецификацию и контроль качества — не угадываешь промптами.

**Цель на перенос (transfer):** студент сам применяет spec-driven цикл к новой задаче / стеку /
инструменту — без инструкции. Это главный признак, что научили, а не натаскали.

**Сквозные ключевые вопросы** (вешаем в W1, возвращаемся на чекпоинте и защите):
- Как заставить ИИ сделать ровно то, что нужно — и доказать, что он это сделал?
- Где ИИ доверять, а где контролировать?

## 1. Программа (канон, 8 недель)

Формат: живые воркшопы, один поток. Повторяете за экспертом + параллельно собираете
**сквозной проект** (реальная/учебная задача: фича / багфикс / рефакторинг). Каждая неделя =
шаг проекта. Финал — репозиторий с полным циклом + защита перед экспертом.

| Нед. | Тема | Что осваивает | Артефакт недели |
|---|---|---|---|
| **W1** | Spec-Driven Development: почему промпты ломаются | модель сбоя агента (контекст/доверие/хаос/качество); **ментальная модель «менеджер отделов»** (что отдать ИИ, что проверять); зачем spec-driven | карта зон доверия/контроля + выбор темы проекта |
| **W2** | От задачи к спецификации | формализация задачи в спеку (требования, ограничения, AC, DoD, тесты) | спецификация (5 секций) |
| **W3** | Управление контекстом проекта | контекст-файл для агента (архитектура, конвенции, границы) | AGENTS.md / CLAUDE.md |
| **W4** | Постановка задач AI-агенту | план работы агента, scope, контрольные точки | agent plan + PR по спеке |
| **W5** | Контроль качества AI-кода | ревью AI-кода по чек-листу, ловля типовых дефектов | заполненный чек-лист ревью + исправленный PR |
| **W6** | Тестирование AI-разработки | тест-стратегия, негативы, coverage как критерий | тесты + coverage-отчёт, зелёный CI |
| **W7** | Работа с легаси и существующим кодом | изменение через спеку в чужом коде без сноса смежного | PR в легаси-репо |
| **W8** | Сквозной проект и защита | сборка полного цикла + демонстрация управления ИИ | репозиторий + защита |

Полный цикл (Экран 5 ленда): **задача → спецификация → контекст → реализация → тесты → ревью → доработка → защита**.

## 2. Образовательные результаты программы (terminal objectives)

Что выпускник умеет **делать** после курса. Каждый — ABCD + компонент 4C/ID + связь с рабочей задачей.

**ОР-1 — Полный spec-driven цикл `[LT]`**
- **A** выпускник · **B** проводит задачу (фича/багфикс/рефактор) через полный spec-driven цикл, управляя AI-агентом · **C** в репозитории, на защите перед экспертом · **D** доходит до прохождения всех критериев готовности без ручного переписывания сгенерённого кода.
- ← Слепок рынка: интеграция всех задач junior+middle.

**ОР-2 — Формализация задачи в спецификацию `[PI]`**
- **A** выпускник · **B** превращает задачу в спецификацию (требования, ограничения, acceptance criteria, DoD, тесты) · **C** для реальной/учебной задачи по шаблону · **D** спека покрывает все 5 секций и однозначно ограничивает агента.
- ← junior task 1 (первичная спека), middle task 1 (полная спека).

**ОР-3 — Управление контекстом `[PI]`**
- **A** выпускник · **B** готовит контекст проекта (архитектура, конвенции, границы изменений) · **C** через AGENTS.md/CLAUDE.md в репозитории · **D** агент выполняет задачу не выходя за границы и не трогая запрещённые файлы.
- ← junior task 2 (контекст для агента), middle task 2/3.

**ОР-4 — Постановка и ведение агента `[LT]`**
- **A** выпускник · **B** ставит задачу агенту в scope и ведёт по плану с контрольными точками · **C** на ограниченном scope в репозитории · **D** получает diff/PR, соответствующий спеке; лишние изменения отклонены.
- ← junior task 3, middle task 2.

**ОР-5 — Контроль качества AI-кода `[PP]`**
- **A** выпускник · **B** проверяет AI-код тестами и чек-листом ревью · **C** имея diff и спеку · **D** выявляет ≥80% типовых дефектов (галлюцинации, лишние правки, тех-долг) до мерджа.
- ← junior task 4, middle task 4/5.

**ОР-6 — Применение в легаси и встраивание в процесс `[LT]`**
- **A** выпускник · **B** безопасно применяет ИИ в существующем/коммерческом коде и встраивает spec-driven в личный/командный workflow · **C** в легаси-репозитории, не им написанном · **D** изменение проходит тесты, не ломает смежное, ограничено scope; процесс повторяем и не завязан на конкретный инструмент.
- ← middle task 3 (управление в существующей базе).

## 3. Образовательные результаты по неделям (ABCD)

| Нед. | A | B (наблюдаемый глагол) | C (условие) | D (критерий) | 4C/ID |
|---|---|---|---|---|---|
| W1 | студент | объясняет, почему промпты ломаются, и классифицирует сбой агента | на 4 кейсах диалога с ИИ | верно атрибутирует причину в ≥3 из 4 | SI |
| W2 | студент | составляет спецификацию (5 секций) | для задачи своего проекта | секции покрыты; результат с/без спеки измеримо расходится | PI·LT |
| W3 | студент | готовит контекст-файл | в своём/учебном репо | агент не выходит за границы, не трогает запретное | PI |
| W4 | студент | ставит задачу в scope и ведёт агента по плану | на атомарной задаче | diff соответствует спеке, лишнее отклонено | LT |
| W5 | студент | прогоняет AI-код по чек-листу ревью | на diff с подсаженными дефектами | находит ≥80% дефектов до мерджа | PP·PI |
| W6 | студент | проектирует тест-стратегию, использует coverage как критерий | для AI-изменений проекта | критичная логика покрыта, негативы есть, CI зелёный | PP |
| W7 | студент | вносит изменение через спеку в чужой код | в легаси-репо | тесты зелёные, смежное не сломано, scope соблюдён | LT |
| W8 | студент | демонстрирует полный цикл и управление ИИ | на защите, репо открыт | проходит рубрику защиты (6 критериев) | LT |

## 4. Карта компетенций (формат вкладки «3. Карта компетенций курса»)

Компетенция = способность успешно решать профессиональную задачу. Складывается из знаний, навыков, инструментов.

| Нед. | Компетенция (TASK) | Знания (KNOWLEDGE) | Навыки (HARD SKILLS) | Инструменты (INSTRUMENTS) |
|---|---|---|---|---|
| W1 | Отличать управляемую AI-разработку от хаотичного промптинга | причины сбоя агента; границы применимости ИИ | классификация сбоя; решение «доверить/проверить» | — |
| W2 | Переводить задачу в спецификацию | структура спеки; AC, DoD, user story | написание спеки; декомпозиция требований | Markdown-шаблон, Gherkin/Given-When-Then, Spec Kit |
| W3 | Управлять контекстом для агента | как агент использует контекст; границы изменений | составление контекст-файла; задание запретных зон | AGENTS.md, CLAUDE.md, .github/copilot-instructions.md |
| W4 | Ставить и вести AI-агента | планирование работы агента; scope, контрольные точки | декомпозиция в план; контроль diff; коррекция | Cursor, Claude Code, Copilot, agent mode, git diff/PR |
| W5 | Контролировать качество AI-кода | типовые дефекты агента; критерии качества; secure coding | ревью по чек-листу; ловля галлюцинаций и тех-долга | review checklist, Copilot/Codex review, ESLint/Ruff |
| W6 | Тестировать AI-разработку | test pyramid; негативные сценарии; coverage | проектирование тестов; настройка CI | PyTest/Jest/JUnit/Vitest/Playwright, GitHub Actions, coverage |
| W7 | Применять ИИ в легаси | риски изменений в чужом коде; предотвращение тех-долга | безопасный рефактор/багфикс через спеку | git branches/PR, dependency graph, ADR, Mermaid |
| W8 | Управлять spec-driven разработкой целиком | весь цикл; как доказать управление | сборка проекта; защита; рефлексия | репозиторий, README, рубрика защиты |

**Soft skills (сквозные):** постановка задач, критическое ревью, рефлексия «где агент додумал», работа по обратной связи.

## 6. Домашние задания (шаг сквозного проекта)

ДЗ каждой недели = следующий шаг своего проекта. Не оторванные упражнения — накопительно к финалу.
Оценка дуальная: **принято / возврат на доработку** (шаблон вкладки 4.1) + уровень T1/T2/T3.

| Нед. | ДЗ | Критерий «принято» | ~Время |
|---|---|---|---|
| W1 | карта доверия/контроля + выбрать тему проекта | тема валидна (фича/багфикс/рефактор), карта заполнена | 1–2 ч |
| W2 | спека на первую задачу проекта | 5 секций, зафиксировано расхождение с/без | 2–3 ч |
| W3 | контекст-файл проекта | агент не трогает запретные файлы | 2 ч |
| W4 | провести задачу агентом по плану | PR соответствует спеке | 3 ч |
| W5 | ревью своего PR по чек-листу | дефекты найдены и устранены | 2 ч |
| **Чекпоинт** (конец W5 / нач. W6) | self + peer-review проекта по чек-листу → светофор 🟢/🟡/🔴; ментор только на 🟡/🔴 | спека/контекст/PR/ревью на месте, scope реалистичен | 0.5 ч |
| W6 | тесты + CI на изменение | критичная логика покрыта, CI зелёный | 3 ч |
| W7 | изменение в легаси через спеку | тесты зелёные, смежное цело | 3–4 ч |
| W8 | финализация репо + защита | рубрика §defense | 4 ч |

> **Промежуточный чекпоинт (W5–6)** — формативная проверка проекта (self + peer-review, светофор).
> Цель: скорректировать траекторию до финала и сэкономить ревью (ментор только на 🟡/🔴). Детали и
> чек-лист — `02-materials/midcourse-checkpoint.md`.

## 7. Опционально (nice-to-have, вне обязательного канона)
- **Дизайн-система как спека для UI** — не отдельная неделя. Опциональный трек для frontend/fullstack:
  врезка в W3–W4 для UI-задач (токены/компоненты/правила = контекст для агента → консистентный UI).
- **Менеджер отделов** — встроен в канон (W1) как ментальная модель.
- **Enrichment / v2:** безопасность AI-разработки; локальные модели / оптимизация стоимости;
  governance на команду (мульти-автор спеки, PR-ревью спек) — из бенчмарка/опроса.
