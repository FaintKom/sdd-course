LXD Kickoff-kit · для обсуждения с экспертом (SME)

Управляемая разработка с ИИ — методический набор для старта

Полный описательный набор, чтобы согласовать курс с экспертом ДО разработки: на чём строим решения (бенчмарк рынка), кого учим, чему научим (образовательные результаты), как устроен путь студента, блюпринты всех 8 модулей с конкретными примерами заданий, оценка — и список решений, которые нужно подтвердить с SME.

8 недель · воркшопы ЦА: junior+ → senior/lead backbone: Specify→Plan→Tasks→Implement→Verify козыри: легаси + капстоун

0Как читать этот документ

Это kickoff-kit — методическая основа для встречи с экспертом. Решения приняты на фактах: источник правды (гугл-таблица курса, 24 вкладки) + опрос студентов + бенчмарк того, как SDD преподают другие (Spec Kit, DeepLearning.AI, Udemy, Kiro, Tessl, CodePath). Зелёным ADOPT помечено проверенное у конкурентов, что берём; красным DIFF — наши отличия (пробелы рынка). Раздел §9 — что нужно от SME. Полный внутренний LXD-дизайн (нагрузка, run-of-show, риски) — в `sdd-course-design.html`.

1Бенчмарк рынка → решения

Все серьёзные источники учат один backbone: Constitution/Steering → Specify → (Clarify) → Plan → Tasks → Implement → Verify. Инструменты учат механике, курсы — смене мышления. Почти всё — короткое, self-paced, видео+квиз, один проект. Полный отчёт: `00-source-materials/competitor-benchmark.md`.

  • EARS-нотация для acceptance criteria (W2): WHEN [событие] THE SYSTEM SHALL [поведение].
  • Clarification gate (NEEDS CLARIFICATION) до планирования.
  • Constitution/steering + AGENTS.md/CLAUDE.md (наш W3).
  • Mindset «power inversion» + lineage TDD/BDD→SDD (W1).
  • Проект, растущий фича-за-фичей с репланом; поурочные квизы.
DIFFERENTIATE — наши козыри (пробелы рынка)

Вывод
Наш 8-недельный канон (из ленда) уже совпадает с индустриальным backbone и закрывает 2 главных пробела рынка (легаси, капстоун). Точечные усиления из бенчмарка: EARS + clarification gate в W2, блок «когда SDD избыточен», governance-на-команду для senior.

2Персона · гейт · тиры

Персона (из опроса + концепции)

  • Коммерческий/проектный разработчик: junior+ → senior/lead; +QA automation, tech lead, тех-фаундер.
  • Роли: backend/frontend/mobile/devops; стек JS/TS, Python, Java/Kotlin, Go, C#.
  • ИИ юзает ежедневно (Cursor/Claude/Copilot/ChatGPT), но хаотично.
  • Боль: агент сносит код, не знает контекст, галлюцинирует, нет системы.

Гейт — по способности пройти петлю

Режем не по «вкусу к качеству» (это результат курса), а по механике участия: читает чужой код · Git (ветки/PR) · запускает тесты · ставит задачу · кодит на 1+ языке. Анти-портрет: не кодит, не знает Git, «приложение без кода».

Тиры (scaffolding fade) — одна полоса, три глубины

РычагT1 junior+T2 middle (ядро)T3 senior/lead
Опорашаблон + worked exampleпустой шаблон + чек-листбез шаблона, только критерий
Проектучебныйсвой рабочий кодсвой легаси + ограничения
Критерийпроцесс пройден+ доказан эффект+ командный стандарт

Expertise reversal: сениору worked example избыточен — стартует с T3. Тир — стартовая точка, не клетка.

3Потребность (gap-анализ)

Разрыв: разработчики юзают ИИ хаотично — задача общими словами, большой кусок кода, нет контроля. Корень — задача плохо формализована, а не «ИИ плохо пишет». Тренинг решает (skill/knowledge-gap: формализация, контекст, проверка). Барьеры среды (setup, платные токены) — не контентом, а pre-work + бесплатный путь.

4Образовательные результаты

Формат ABCD (Audience·Behavior·Condition·Degree). Полные формулировки — `program-and-outcomes.md`.

Уровень программы (что выпускник умеет делать)

  1. ОР-1: проводит задачу через полный spec-driven цикл, управляя агентом → все критерии готовности без ручного переписывания. DIFF капстоун
  2. ОР-2: формализует задачу в спецификацию (требования, ограничения, EARS-AC, DoD, тесты). EARS
  3. ОР-3: управляет контекстом (constitution/AGENTS.md/CLAUDE.md) → агент не выходит за границы.
  4. ОР-4: ставит и ведёт агента в scope с контрольными точками → diff соответствует спеке.
  5. ОР-5: проверяет AI-код тестами и ревью → ловит ≥80% типовых дефектов.
  6. ОР-6: применяет ИИ в легаси и встраивает процесс в команду. DIFF легаси

6Блюпринты модулей + примеры заданий

На каждый модуль: результат, что учим, метод/инструмент (с пометкой бенчмарка), конкретный пример задания, артефакт, проверка. Примеры заданий — благодатные паттерны из бенчмарка, адаптированные.

W1Почему промпты ломаютсяMindset
Учим

«Power inversion»: код служит спеке. Lineage TDD/BDD→SDD. 4 типа сбоя: контекст/доверие/хаос/качество. Udemy M1

Пример задания
Атрибуция сбоя: дано 4 диалога «промпт → результат агента». Определи тип сбоя в каждом и привяжи к неделе-лечению (контекст→W3, доверие→W5/6, хаос→W2, качество→W5). Демо: напиши мини-конституцию, нарушь её, посмотри, как агент дрейфует. Spec Kit gate-demo
Артефакт / проверка

Карта доверия/контроля + тема проекта. ≥3 из 4 кейсов верно.

W2От задачи к спецификацииSpecify + Clarify
Учим

Спека: цель, требования, ограничения, EARS-AC, DoD, тесты. Clarification gate. EARS · NEEDS CLARIFICATION

Пример задания
Vague → EARS: дана размытая user story «хочу отмену заказа». Перепиши требования в EARS (WHEN отмена в окне 30 мин THE SYSTEM SHALL отменить заказ), помечай неясное NEEDS CLARIFICATION, затем сгенерь тест-кейсы прямо из EARS-критериев. +A/B: прогон агента с/без спеки, зафиксируй расхождение. Kiro/Udemy
Артефакт / проверка

Спека (5 секций, EARS). Все секции; измеримое расхождение с/без.

W3Управление контекстом (constitution)Steering
Учим

Constitution/steering: AGENTS.md/CLAUDE.md, «always / ask first / never do», границы изменений. Spec Kit · Udemy M3

Пример задания
Конституция репо: напиши AGENTS.md для своего репо с «always/ask/never» и списком запретных файлов. Затем намеренно опусти одну границу и прогони задачу — увидь, как агент полезет лишнее. Добавь границу, повтори. DLAI lesson 6
Артефакт / проверка

AGENTS.md/CLAUDE.md. Агент не трогает запретные файлы.

W4Постановка задач агентуPlan + Tasks + Implement
Учим

План → атомарные задачи (с зависимостями) → implement human-in-the-loop, контрольные точки. Spec Kit tasks · Kiro waves

Пример задания
Quick vs gated A/B: проведи одну задачу «быстрым планом» без гейтов и ту же — через план+tasks с контрольными точками. Сравни diff, лишние правки, переделки. Прочувствуй цену пропуска гейтов. Kiro Quick Plan
Артефакт / проверка

Agent plan + PR. Diff соответствует спеке, лишнее отклонено.

W5Контроль качества AI-кодаVerify · review
Учим

Чек-лист ревью (соответствие/границы/галлюцинации/тех-долг/безопасность). Типовые ошибки агента.

Пример задания
Охота на дефекты: дан PR с 10 подсаженными дефектами (выдуманный API, лишний файл, заглушка, захардкоженный ключ, скрытый регресс <= → <, фейк-тест...). Найди по чек-листу, верни агенту точечно. Цель: T1 ≥3/4, T2 ≥6/8, T3 ≥8/10. наш набор
Артефакт / проверка

Заполненный чек-лист + исправленный PR. Детект ≥80%.

W6Тестирование AI-разработкиVerify · tests
Учим

Тесты из спеки (не после кода), негативы, coverage как критерий, CI. tests-as-spec

Пример задания
EARS → тесты: возьми EARS-критерии из своей спеки (W2) и сгенерь из них набор тестов (happy + негатив + edge), прогони в CI, добей покрытие критичной логики. Kiro/Udemy
Артефакт / проверка

Тесты + coverage-отчёт, зелёный CI. Негативы есть.

W7Легаси и существующий кодBrownfield · DIFF
Учим

Ретрофит SDD на чужой грязный репо без сноса смежного; предотвращение тех-долга. главный козырь — у конкурентов 0–1 урок

Пример задания
Ретрофит на легаси: дан учебный легаси-репо с тестами и «минами» (хрупкие зависимости). Напиши спеку на багфикс/фичу, конституцию для этого репо, проведи агента — изменение проходит тесты и не ломает смежное. T3: свой рабочий легаси + бюджет токенов как ограничение.
Артефакт / проверка

PR в легаси-репо. Тесты зелёные, scope соблюдён.

W8Сквозной проект и защитаCapstone · DIFF
Учим

Сборка полного цикла; как доказать управление ИИ, не только итог. graded капстоун + peer review

Пример задания
Капстоун: отгрузи фичу с полным следом задача→спека(EARS)→контекст→tasks→implement→тесты→ревью→доработка. Защита 5–7 мин: покажи, как спека удержала агента, что поймало ревью. Peer review спеки сокурсника. IBM/OpenAI capstone
Артефакт / проверка

Репозиторий + защита. Рубрика (6 критериев × T1–T3) — `02-materials/defense-rubric.md`.

Опц. enrichment (senior-трек / v2, из бенчмарка)
Governance на команду (мульти-автор спеки, PR-ревью спек) · drift detection / жизнь спеки (идея Tessl) · честный блок «когда SDD избыточен» (review fatigue, Fowler) · architecture-decision спеки + contract testing. Из опроса (вне канона): безопасность AI-разработки, локальные модели/стоимость.

8Оценка

Kirkpatrick → метрики групп

  • L1 реакция — NPS, балл занятия ≥4.5
  • L2 обучение — артефакт-критерий/нед + рубрика; ревью-детект ≥80%
  • L3 поведение — опрос 30/60 дн: применил ли spec-driven (≥50%)
  • L4 результат — кейсы внедрения в команду

Leading indicators

  • COR (70% занятий, 50% ДЗ) + доходимость до проекта
  • Ревью-детект W5 — ранний прокси освоения ядра
  • Расхождение с/без спеки W2 — early win

Капстоун оценивается рубрикой защиты (6 критериев × T1/T2/T3), дуально «принято/возврат» + peer review.

9Вопросы и решения к SME

Что подтвердить/заполнить с экспертом до разработки.

Контент и метод
  1. Подтвердить последовательность 8 недель и усиления из бенчмарка: EARS в W2, clarification gate, «когда SDD избыточен». Берём?
  2. Reference-инструмент: учить tool-agnostic на Spec Kit (free) + основной агент потока — Cursor или Claude Code? (бесплатный путь — Copilot/ChatGPT/локальные).
  3. Стек примеров: ленд говорит «Python middle+», но ЦА мульти-стек. Делаем примеры на Python с языко-нейтральными спеками? Или мульти-стек?
  4. Глубина governance-на-команду и drift — в основной курс (senior-трек) или v2?
Практика и проект
  1. Источник учебного легаси-репо (W7) и набора подсаженных дефектов (W5) — даёт эксперт или собираем мы?
  2. Темы сквозного проекта: своя задача студента vs учебный список. Список нужен — кто даёт?
  3. Капстоун: индивидуально или команды 2–3? Peer review спек — включаем?
Орг и роли
  1. Эксперт/спикер курса — кто ведёт воркшопы (повтор за экспертом — ключевой формат)?
  2. Длительность воркшопа: ~2 ч/нед — подтверждаем? (формат-аналог CodePath: 1 live 2ч + 4–9ч дз)
  3. Сертификат участия — ок (не диплом)?
